# Frontend Project Architecture Convention

## 1. Architecture Principles

### 1️⃣ Feature-Oriented Architecture

项目必须以 **Feature（功能模块）为核心组织代码**。

目标：

- 降低模块耦合
- 提高可维护性
- 提高可读性
- 支持未来拆分 **Micro Frontends**

原则：

```
Most application code MUST live inside /features
```

即：

> 绝大部分业务代码必须在 `features` 目录中。

---

# 2. Root Folder Structure

标准 `src` 结构：

```
src
├── assets
├── components
├── composables
├── config
├── features
├── layouts
├── lib
├── pages
├── services
├── stores
├── test
├── types
└── utils
```

## 目录职责

| 目录          | 作用                                    |
| ------------- | --------------------------------------- |
| `assets`      | 静态资源（images / fonts / global css） |
| `components`  | **全局复用组件**                        |
| `composables` | 全局可复用 hooks / composables          |
| `config`      | 应用配置                                |
| `features`    | **核心业务模块**                        |
| `layouts`     | 页面布局                                |
| `pages`       | 页面组件                                |
| `lib`         | 第三方库初始化                          |
| `services`    | 全局服务（API client / providers）      |
| `stores`      | 全局状态                                |
| `test`        | 测试工具                                |
| `types`       | 全局类型                                |
| `utils`       | 工具函数                                |

---

# 3. Pages 规则

`pages` 只负责：

```
Routing + Page Layout
```

**禁止在 pages 中写复杂业务逻辑**

页面应只负责：

```
pages
 └── home
      └── HomePage.tsx
```

示例：

```ts
import { Timeline } from "@/features/timeline"

export default function HomePage() {
  return <Timeline />
}
```

---

# 4. Feature Module Structure

每个 Feature 是 **一个独立模块**。

结构：

```
features
 └── timeline
      ├── api
      ├── components
      ├── composables
      ├── stores
      ├── types
      └── index.ts
```

## 目录职责

| 目录          | 作用             |
| ------------- | ---------------- |
| `api`         | 所有 API 请求    |
| `components`  | Feature 私有组件 |
| `composables` | Feature hooks    |
| `stores`      | Feature 状态     |
| `types`       | Feature 类型     |
| `index.ts`    | Feature 公共 API |

---

# 5. Feature Public API Pattern

每个 Feature 必须有：

```
index.ts
```

它是 Feature 的 **public API**。

示例：

```
features/profile
 ├── components
 │    └── UserProfile.vue
 └── index.ts
```

`index.ts`

```ts
export { default as UserProfile } from "./components/UserProfile.vue";
```

---

# 6. Import Rules

## ❌ 禁止

跨模块直接引用内部文件

```
import { UserProfile } from '@/features/profile/components/UserProfile.vue'
```

---

## ✅ 必须

通过 feature API 引入

```
import { UserProfile } from '@/features/profile'
```

优点：

- 防止循环依赖
- 明确模块边界
- 统一依赖入口

---

# 7. Shared Code Guidelines

理想情况下：

```
All code should live inside features
```

但实际项目中允许共享代码。

共享代码必须放在：

```
components
composables
stores
services
types
utils
```

添加共享代码时必须谨慎：

原则：

```
If code is used by only one feature
→ keep it inside that feature
```

---

# 8. API Layer Separation

所有 API 请求必须在：

```
features/<feature>/api
```

示例：

```
features
 └── tweets
      └── api
           └── getTweets.ts
```

例：

```ts
export async function getTweets() {
  return fetch("/api/tweets");
}
```

UI 不应直接写 API 请求。

---

# 9. State Management Rule

Feature 状态：

```
features/<feature>/stores
```

全局状态：

```
src/stores
```

---

# 10. ESLint Enforcement

必须添加规则：

```js
rules: {
  "no-restricted-imports": [
    "error",
    {
      "patterns": ["@/features/*/*"]
    }
  ],
  "import/no-cycle": "error"
}
```

作用：

1️⃣ 禁止跨 feature 内部引用
2️⃣ 防止循环依赖

---

# 11. Feature Isolation Principle

Feature 必须满足：

```
Feature = mini application
```

特征：

- 独立组件
- 独立 API
- 独立 state
- 独立 types

目标：

```
Feature should be movable without breaking the app
```

---

# 12. Agent Coding Rules (for AI)

Agent 必须遵循：

### 创建新功能

必须：

```
src/features/<feature-name>
```

结构：

```
features
 └── chat
      ├── api
      ├── components
      ├── composables
      ├── stores
      ├── types
      └── index.ts
```

---

### 新组件

判断：

| 情况             | 放置位置                        |
| ---------------- | ------------------------------- |
| 只属于某 feature | `features/<feature>/components` |
| 全局复用         | `src/components`                |

---

### 新 API

```
features/<feature>/api
```

---

### 新 hook

```
features/<feature>/composables
```

---

### 新 store

```
features/<feature>/stores
```

---

# 13. Expected Benefits

该架构可以：

- 降低耦合
- 防止依赖混乱
- 提高代码定位速度
- 方便 onboarding
- 方便未来拆分 **micro frontends**
